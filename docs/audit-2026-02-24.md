## Findings

### Code quality

1. **Info** — Architecture is clean. Four orthogonal functions
   (parse/extract/diff/apply), no circular dependencies, proper separation of
   concerns. Library has no logging dependency; consumers own diagnostics.

2. **Info** — RAII wrappers (Database, Statement) are excellent. Proper move
   semantics, deleted copy constructors, null-safe destructors with correct
   `this != &other` checks in move assignment.

3. **Info** — No dead code, no TODOs/FIXMEs, consistent snake_case naming
   throughout. All functions are reachable and used.

4. **Medium** — `extract()` (~160 lines) and `parse_create_table_body()` (~140
   lines) in sqlift.cpp are the longest functions. Not excessively complex but
   worth noting for future maintainability.

5. **Medium** — `Column::generated` uses magic integers (0=normal, 2=virtual,
   3=stored) inherited from SQLite's `table_xinfo` PRAGMA (sqlift.h:144). An
   enum would be clearer. Decision needed before 1.0 — changing after would be
   a breaking change. Tracked in STABILITY.md.

### Security

6. **Info** — All SQL identifiers are quoted via `quote_id()`, which properly
   escapes embedded quotes. No SQL injection risk from identifier handling.

7. **Info** — `parse()` executes arbitrary DDL by design. This is the intended
   use case for a schema migration library. Consumers provide the SQL.

8. **Info** — SHA-256 implementation is textbook-correct. Initial hash values,
   round constants, bit operations, message schedule, and compression loop all
   match FIPS 180-4. Used for drift detection (integrity, not secrecy).

9. **Info** — No hardcoded secrets, credentials, weak cryptographic algorithms,
   or non-cryptographic PRNGs in security-sensitive contexts.

### Testing

10. **Info** — 83 tests, 275 assertions, all passing. Every public API function
    is tested. Tests run in-memory (`:memory:` databases), single command
    (`mk test`), well-isolated with no shared state or filesystem artifacts.

11. **Info** — Good edge case coverage: FK violations with detailed error
    messages, SAVEPOINT rollback error recovery, partial and expression indexes,
    CHECK/COLLATE/GENERATED/STRICT features, breaking change detection, JSON
    validation on deserialization.

### Performance

12. **Info** — No algorithmic complexity issues. Schema diff is O(n) in the
    number of schema objects. `std::map` provides ordered iteration for
    deterministic output. Proper RAII throughout, no resource leaks.

### Build and CI/CD

13. **Info** — Build works, all tests pass. mkfile is clean and minimal (33
    lines). No CI workflows (appropriate for a library at this stage).

14. **Low** — `.gitignore` does not cover `.claude/` or `.mk/` directories,
    which appear as untracked files. Should be added to prevent accidental
    commits.

### Legal and licensing

15. **Critical** — No third-party attribution file exists. Two vendored
    MIT-licensed dependencies (doctest, nlohmann/json) require their licence
    texts to be distributed with the project per Apache 2.0 Section 4(d). Both
    dependencies have proper licence headers in their vendored files, but a
    consolidated NOTICES or THIRD_PARTY_LICENSES file is missing.

16. **Info** — LICENSE file is present and correct (Apache 2.0, Copyright 2026
    Marcelo Cantos). All source files have matching `SPDX-License-Identifier:
    Apache-2.0` headers. Copyright year is current.

17. **Info** — Both vendored dependencies are licence-compatible. doctest is
    MIT (Viktor Kirilov 2016-2023). nlohmann/json is MIT (Niels Lohmann
    2013-2023). SQLite is public domain (no attribution required).

### Documentation

18. **High** — docs/reference.md is stale. Missing documentation for:
    Column::collation, Column::generated, Column::generated_expr,
    CheckConstraint struct, Table::check_constraints, Table::strict,
    BreakingChangeError exception, and Statement::column_int/bind_int as
    int64_t (docs still show int).

19. **High** — docs/guide.md does not mention: CHECK constraint support,
    COLLATE clause support, GENERATED column support, STRICT table support,
    breaking change detection (BreakingChangeError), view/trigger topological
    sort for dependency ordering, or JSON serialization/deserialization.

20. **Medium** — README.md (line 47) claims "No runtime dependencies beyond
    SQLite3" but nlohmann/json is vendored in vendor/include/nlohmann/ and
    compiled into sqlift.cpp. Should clarify that it is vendored (no external
    install required) rather than absent.

### Dependency health

21. **Low** — nlohmann/json is vendored at v3.11.3. Upstream is 3.13+. No
    known security issues, but worth updating before 1.0 for bug fixes and
    improvements.

22. **Info** — Both vendored dependencies are clean, unmodified copies of their
    upstream releases. All dependencies are justified: doctest for testing,
    nlohmann/json for JSON plan serialization, SQLite3 as the core domain.

### Agent-friendliness

23. **Info** — CLAUDE.md, agents-guide.md, and STABILITY.md are all present and
    useful. Build is discoverable via `mk test`. Project structure is
    conventional (header + implementation + tests/). Error messages include
    SQLite diagnostic text, suitable for automated diagnosis.

### Portability

24. **Info** — No platform-specific code. Uses only standard library containers
    and SQLite3 APIs. C++23 requirement is documented in README.md. Code is
    portable across Linux, macOS, and Windows with a compatible compiler
    (GCC 14+, Clang 17+, MSVC 2022+).

25. **Info** — `sha256()` is declared public in sqlift.h:129 but is an
    implementation detail used only for schema hashing and drift detection.
    Exposing it implies a stability commitment unrelated to schema migration.
    Decision needed before 1.0. Tracked in STABILITY.md.
