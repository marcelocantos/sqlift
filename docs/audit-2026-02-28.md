## Detailed Findings

### T1. HIGH — `apply()` leaves FK enforcement OFF on failed rebuild

**Category**: Code correctness (bug)
**Location**: `sqlift.cpp:869-948`, `tests/test_apply.cpp:129`

When `apply()` executes a table rebuild, the sequence starts with
`PRAGMA foreign_keys=OFF` (step 1). If the FK check at step 10 throws
`ApplyError`, `PRAGMA foreign_keys=ON` (step 12) is never executed. The
SAVEPOINT is rolled back by SQLite, but `PRAGMA foreign_keys` is
connection-level state — it is not transactional. After a failed rebuild,
the caller's connection silently has FK enforcement disabled.

The existing test ("apply error recovery preserves database state") does not
check FK enforcement state after the failure, giving a false sense of safety.

**Fix**: Wrap the rebuild execution in a try/catch that restores
`PRAGMA foreign_keys=ON` before rethrowing. Add test assertions that verify
FK enforcement is ON after both successful and failed rebuilds.

### 5. MEDIUM — SQL parser doesn't handle string literals in paren tracking

**Category**: Code correctness
**Location**: `sqlift.cpp:331-498`

`parse_create_table_body()` tracks parenthesis depth for splitting by `,` and
for matching `CHECK(...)` / `FOREIGN KEY(...)` expressions. It does not skip
content inside string literals (`'...'`, `"..."`). A CHECK constraint like
`CHECK(x = ')')` would miscount depth and mis-parse surrounding structure.

The raw_sql comes from `sqlite_master`, which stores the original DDL, so
unusual but legal SQL could trigger this.

**Fix**: Add string-literal awareness to the depth-tracking loops — skip
content inside `'...'` and `"..."`.

### 1. MEDIUM — Unchecked GeneratedType cast

**Category**: Code correctness
**Location**: `sqlift.cpp:579`

`static_cast<GeneratedType>(col_stmt.column_int(6))` trusts SQLite's
`table_xinfo` to return only 0, 2, or 3. A future SQLite version could
introduce new values, silently creating an invalid enum.

**Fix**: Validate the value after the cast; throw `ExtractError` for
unrecognised values.

### T5. MEDIUM — No direct test for Schema::hash()

**Category**: Test coverage
**Location**: `sqlift.cpp:213`

`Schema::hash()` is critical for drift detection but has no direct test. A
regression in hash serialization could produce collisions or non-determinism,
with silent failures.

**Fix**: Add tests verifying: same schema = same hash (determinism), and
distinct schemas = distinct hashes (for common change types).

---

### T7. MEDIUM — Constraint name in hash but not equality creates permanent drift

**Category**: Design inconsistency
**Location**: `sqlift.cpp:232-236`

Constraint names are excluded from `operator==` (cosmetic) but included in
`Schema::hash()`. If a constraint name changes (e.g., via raw SQL outside
sqlift), `diff()` produces an empty plan (schemas are structurally equal) but
the hash changes. On next `apply()`, the stored hash won't match the actual
schema hash, causing a permanent `DriftError` with no way to resolve it via
migration.

**Fix**: Either exclude constraint names from the hash (matching equality
semantics), or include them in equality (so `diff()` produces a rebuild plan
that resolves the discrepancy). The former is simpler.
